## CDX-010 — capability-language rewrite: the 9 `AskUserQuestion` skills (§7.1)

**§7.1 exact text**: "WHERE a skill needs structured user input THE SYSTEM SHALL describe it in capability language ('ask through the host's structured-input facility when available; otherwise ask one concise direct question') in the shared `SKILL.md`, with any exact tool call isolated to a `references/host-claude.md` adapter."

**Glossary definitions** (§4): **Capability language** — "skill prose that describes what must happen ('ask the user through the host's structured-input facility') instead of naming an exact host tool (`AskUserQuestion`)." **Host adapter** — "the smallest isolated piece of a skill (a `references/host-<name>.md` doc...) that differs per host; everything else is shared."

**Canonical phrase** (this task establishes it — first host-adapter work in the repo, no prior precedent to match): replace "AskUserQuestion" in shared prose with **"the host's structured-input facility"**, matching §7.1's own worked example verbatim. Where the current text says "AskUserQuestion (header 'X')" write "the host's structured-input facility (header 'X')". Where it says "call AskUserQuestion" write "invoke the host's structured-input facility" or "ask through the host's structured-input facility" (vary naturally per sentence, don't robotically find-replace).

**Inventory — exactly 9 `SKILL.md` files** (confirmed via `grep -rl AskUserQuestion plugins/spec-workflow/skills/*/SKILL.md`): `setup-project`, `auto-merge`, `build-next`, `pr-review-model`, `agent-identities`, `concurrency`, `craft-spec`, `ask-identity`, `create-inbound`.

**`allowed-tools` frontmatter is OUT OF SCOPE** — per §12's own invariant ("`name` and `description` are the only frontmatter fields either host is assumed to enforce; `allowed-tools` remains a Claude-only enhancement, never the cross-host security boundary"), the frontmatter `allowed-tools: ... AskUserQuestion ...` lines (present in `auto-merge`, `pr-review-model`, `agent-identities`, `concurrency`, `ask-identity`) are left UNCHANGED. Only prose BODIES are rewritten, matching the acceptance criterion's own wording ("shared `SKILL.md` bodies").

**Adapter strategy — simple (inline) vs complex (dedicated file)**, per the task's own "only where the skill's complexity warrants a separate file; simple cases may inline a one-line Claude note":
- **Complex → dedicated `references/host-claude.md`**: `craft-spec` (an interview LOOP — "rounds of at most 4 questions using the question bank," plus a separate sign-off loop — enough structure to warrant its own file) and `setup-project` (THREE distinct structured-input moments across different phases — new/existing project choice, merge policy with a follow-up sub-question, feedback opt-in — multi-decision-point complexity).
- **Simple → inline one-line Claude note**, appended near the (now capability-language) mention: `auto-merge`, `build-next`, `pr-review-model`, `agent-identities`, `concurrency`, `ask-identity`, `create-inbound`. Inline note pattern: `(On Claude Code, this is the AskUserQuestion tool.)` — placed once per skill, near its first capability-language mention, not repeated at every occurrence within the same file.

**Constraints to preserve verbatim** (acceptance criterion: "constraints... are preserved"), per skill — from the research inventory:
- `craft-spec`: max 4 questions per interview round; the sign-off loop iterates until approved.
- `auto-merge`: "put the CURRENT state's opposite first" (option-ordering rule, state-dependent) for its first ask; a second, simpler 2-option ask for pre-authorization.
- `ask-identity`: exactly 3 options, exact wording ("It is" / "It is not" / "I'm unsure").
- `create-inbound`: stop/continue semantics — "If the human is absent or does not answer, do NOT create."
- `pr-review-model`: 4 options + a free-text "Other" affordance, no previews.
- `agent-identities`, `concurrency`: 3 options each, current-value-noted framing.
- `setup-project`: 3 distinct asks (new/existing project; merge policy + sub-question; feedback opt-in with a Recommended default).
- `build-next`: the session-consent-gate single-ask (§0 of auto-review.md) AND the separate NEGATIVE operating rule ("does not use [the structured-input facility] unless a hard permission denial or an explicit instruction requires human direction") — both must survive the rewrite, the second one is not a request-input call at all, just a behavioral constraint mentioning the facility by (now capability) name.

**Existing test needing rewrite** (not just addition): `plugins/spec-workflow/tests/section-skill-contracts.sh:71` currently asserts the LITERAL string `"does not use AskUserQuestion unless a hard permission denial..."` is present in build-next's body — this exact assertion will start failing once that sentence is rewritten in capability language, by design (that's the whole point of the acceptance criterion: "none of the 9 shared SKILL.md bodies require a tool literally named AskUserQuestion to function"). Update this existing check's expected string to match the new capability-language wording — do not just delete the check, keep the same operating-rule constraint covered.

**New tests**: for the 2 complex skills, assert their new `references/host-claude.md` file exists and contains the preserved constraint (e.g. craft-spec's "4 questions" number, the exact tool name `AskUserQuestion` — the adapter file is exactly where that literal name is SUPPOSED to still live). For all 9 skills' `SKILL.md` bodies, assert the literal string `AskUserQuestion` is ABSENT (this is the core acceptance criterion, directly testable) while the capability-language phrase IS present.

## CDX-011 — plan-mode / no-write phase as capability language (§7.2)

**§7.2 exact text**: "WHERE a skill has a no-write research/design phase (e.g. `craft-spec`'s Phase 1) THE SYSTEM SHALL express it as an explicit behavioral constraint ('no file writes during discovery') rather than solely as a call to a Claude-specific plan-mode tool."

**Inventory — only `craft-spec/SKILL.md` mentions `EnterPlanMode`/`ExitPlanMode`** (confirmed: `grep -rl "EnterPlanMode\|ExitPlanMode" plugins/spec-workflow/skills/` returns exactly one file). Two occurrences:
- Line 17 (Phase 1 — discover): "Enter plan mode (EnterPlanMode) if available: this phase is research + design, no file writes yet." — the behavioral constraint ("no file writes yet") is ALREADY stated in the same sentence, just conjoined with the Claude-specific tool call rather than led by it.
- Line 29 (Phase 4 — review sign-off): "...via plan approval (ExitPlanMode) if in plan mode, else directly." — this is the exit/approval action, already largely capability-language (the actual sign-off ask already uses "the host's structured-input facility" per CDX-010's rewrite) with only the `(ExitPlanMode)` parenthetical needing adapter treatment.

**Rewrite plan**: lead each sentence with the BEHAVIORAL CONSTRAINT ("no file writes during discovery/design" for line 17; "present a summary and get sign-off" for line 29), with the Claude-specific tool name moved into the EXISTING `references/host-claude.md` adapter (already created by CDX-010 for this same skill's structured-input mechanics — this task extends that same file with a new "Phase 1 / no-write phase" section, and a note on Phase 4's approval mechanics) rather than inline in the shared body, since this skill already has a dedicated adapter file and craft-spec was independently classified "complex" enough for one in CDX-010.

**Constraint to preserve**: "no file writes during discovery/design" (Phase 1's substance) must survive as an explicit, host-agnostic, independently-testable behavioral statement — not just implied by a tool-call instruction. craft-spec's actual Claude behavior (entering/exiting plan mode) must be unchanged — the adapter still instructs a Claude-side agent to call `EnterPlanMode`/`ExitPlanMode`.

**New tests**: assert `craft-spec/SKILL.md`'s body states the "no file writes" constraint in prose not gated on a tool name (a `check` for a host-agnostic phrase like "no file writes" or "no-write phase"); assert the body does NOT require the literal strings `EnterPlanMode`/`ExitPlanMode` to understand the instruction (mirror CDX-010's absent-from-body pattern); assert `references/host-claude.md` contains both literal tool names and documents when to call them.

## CDX-012 — delegation-spawn capability language (§7.3)

**§7.3 exact text**: "WHERE a skill delegates bounded work to a subagent THE SYSTEM SHALL describe the delegation in capability language ('delegate to a fresh implementation agent when the host supports delegation') with exact spawn parameters (tool name, `subagent_type`) isolated to a host adapter."

**Inventory — exactly 2 sites repo-wide** (confirmed via `grep -rn "Agent tool\|subagent_type" plugins/spec-workflow/`, the full plugin tree, not just `skills/*/SKILL.md`):
- `implement-task/SKILL.md:25` — dev-agent spawn instruction: "Spawn with the Agent tool, `subagent_type: general-purpose`, `model: <...>`, and `name: dev-<task-id>` (...)".
- `build-next/references/auto-review.md:65` — reviewer-agent spawn instruction: "spawn ONE reviewer agent — Agent tool, `subagent_type: general-purpose`, `model:` a suitable id from the reviewer identity's allowed set (...)".

No other skill or reference file names `Agent tool` or `subagent_type`. `build-next/SKILL.md`'s own body (step 2: "spawn a dev subagent...") already uses host-neutral phrasing with no literal tool name — it needs no rewrite; the acceptance criterion's mention of "build-next" refers to its reference doc, `auto-review.md`, where the actual spawn-parameter prose lives.

**Out of scope (feeds CDX-02x instead)**: model SELECTION (`identity.sh dev`'s allowed-model set, choosing a suitable id) is unchanged — §7.3 only governs the SPAWN mechanism's tool/parameter naming, not which model is picked. `name: dev-<task-id>` / `name: pr-reviewer-<pr-number>` naming conventions are host-neutral already (just a label) and are preserved verbatim.

**Adapter strategy — both sites are simple (exactly one literal mention each)**, matching CDX-010's own classification rule ("simple cases may inline a one-line Claude note"): no new `references/host-claude.md` file for either skill. Instead, inline a one-line Claude note immediately after the (now capability-language) spawn sentence, mirroring CDX-010's exact inline-note pattern:
- `implement-task/SKILL.md:25` — replace "Spawn with the Agent tool, `subagent_type: general-purpose`, `model: <the id you chose from that allowed set>`, and `name: dev-<task-id>` (...)" with "Delegate to a fresh implementation agent when the host supports delegation, with `model: <the id you chose from that allowed set>` and `name: dev-<task-id>` (...)" followed by "(On Claude Code, this is the Agent tool with `subagent_type: general-purpose`.)".
- `build-next/references/auto-review.md:65` — replace "spawn ONE reviewer agent — Agent tool, `subagent_type: general-purpose`, `model:` a suitable id from the reviewer identity's allowed set (...)" with "delegate to ONE fresh reviewer agent when the host supports delegation, with `model:` a suitable id from the reviewer identity's allowed set (...)" followed by the same inline Claude note.

**Constraints to preserve verbatim**: `implement-task`'s naming rule (`dev-<task-id>`, role-prefix first, re-brief appends a letter, never a bare counter) and its "one agent = one task" rule; `auto-review.md`'s naming rule (`pr-reviewer-<pr-number>`) and "keep this SAME agent for every round... via SendMessage" continuity rule (`SendMessage` itself is out of scope for this task — §7.3 governs the SPAWN action only, not the follow-up dialogue mechanism; a future task may extend capability language to it).

**New tests**: extend `section-capability-language.sh` with a CDX-012 block mirroring the existing simple-skill pattern — for each of the 2 files, assert the capability-language phrase ("delegate to a fresh implementation agent when the host supports delegation" / "delegate to ONE fresh reviewer agent when the host supports delegation") is present, `Agent tool`/`subagent_type` appears exactly once each (the inline Claude note), the inline-note text itself is present, and every preserved constraint (naming rules, one-agent-per-task, SendMessage continuity) survives verbatim.

## CDX-013 — argument-semantics audit (§7.4, §7.5)

**§7.4 exact text**: "THE SYSTEM SHALL NOT reference an `ARGUMENTS:`-style variable in shared skill prose as if it were guaranteed to be populated; any such prose SHALL treat skill arguments as the remainder of the user's request text."

**§7.5 exact text**: "WHEN a skill's pre-start step is currently expressed as Claude command-substitution prose (e.g. `!`-prefixed shell) THE SYSTEM SHALL convert it into an explicit first workflow step usable by either host."

**Inventory — exactly 7 sites, 2 patterns, confirmed repo-wide** (`grep -rln "ARGUMENTS:" plugins/*/skills/*/SKILL.md` and `grep -rln '!\`' plugins/*/skills/*/SKILL.md` across every plugin in the repo, not just spec-workflow — the other two plugins, `peer-review` and `scaffold-project`, have zero hits of either pattern, consistent with SPEC-CODEX-COMPAT.md's scope being the spec-workflow plugin specifically):

- **§7.4 (`ARGUMENTS:`), 2 files**: `ask-brain/SKILL.md:9` — `"ARGUMENTS: \`<question...>\` — the question, verbatim."`; `ask-identity/SKILL.md:9-11` — `"ARGUMENTS: \`<identity> <question...>\` — the first token names the role (\`dev\`, \`reviewer\`, \`orchestrator\`, or a repo-specific custom role like \`judge\`/\`player\`), everything after it is the question, verbatim."`
- **§7.5 (`!`-prefixed pre-start check), 5 files, IDENTICAL line**: `build-next/SKILL.md:9`, `implement-task/SKILL.md:9`, `next-task/SKILL.md:9`, `queue/SKILL.md:11`, `seed-board/SKILL.md:9` — all read `Pre-start check: !\`bash "../../scripts/preflight.sh" --spec\`` followed by `If the line above says \`PREFLIGHT FAIL\`, STOP — follow its instruction instead of continuing.` (line-for-line identical across all 5; `queue`'s is at line 11 because it has 2 extra header lines, not a different pattern).

**Why these are Claude-specific mechanics**: Claude Code's CLI evaluates a `!`-prefixed backtick span as shell command substitution BEFORE the model ever sees the prompt — it runs the command and splices the raw stdout inline in place of the `!\`...\`` token (verified empirically: this is exactly what happens on every real invocation of these 5 skills this session — the model sees `Pre-start check: preflight ok: ...` as literal prompt text, never the `!\`...\`` syntax itself). A Codex-side agent has no equivalent CLI-level substitution feature, so the literal `!\`...\`` token would reach it un-evaluated and be meaningless prose. `ARGUMENTS:` is a similar CLI-populated convention (Claude Code slash commands receive `$ARGUMENTS`/positional interpolation from the CLI); referencing it as a labelled header assumes a host mechanism that fills it in — Codex has no guaranteed equivalent, so the skill body must describe the SAME semantic (arguments = whatever text followed the command invocation) without depending on a host to have already parsed/populated it.

**Rewrite plan**:
- **§7.5 sites** — replace the two-line `Pre-start check: !\`...\`` / `If the line above says...` pair with a single explicit workflow step the agent performs itself, in BOTH hosts identically: `Pre-start check — run this now, before anything else: \`bash "../../scripts/preflight.sh" --spec\`. If it prints \`PREFLIGHT FAIL\`, STOP — follow its instruction instead of continuing.` This is a mechanical, identical find-replace across all 5 files (the command path is always `"../../scripts/preflight.sh" --spec`, no per-skill variation to preserve).
- **§7.4 sites** — replace `ARGUMENTS: ...` header lines with prose treating the arguments as "the remainder of the user's request text" (§7.4's own worked phrase), preserving each skill's exact parsing semantics:
  - `ask-brain`: `"Treat the remainder of the user's request (after the command name) as the question, verbatim."`
  - `ask-identity`: `"Treat the remainder of the user's request (after the command name) as \`<identity> <question...>\`: the first token names the role (\`dev\`, \`reviewer\`, \`orchestrator\`, or a repo-specific custom role like \`judge\`/\`player\`), everything after it is the question, verbatim."`

**Constraints to preserve verbatim**: `ask-identity`'s exact role-name examples (`dev`, `reviewer`, `orchestrator`, `judge`/`player`) and the "first token = role, rest = question" parsing rule; `ask-brain`'s "question, verbatim" framing; every §7.5 site's `PREFLIGHT FAIL` stop-condition wording and the exact preflight command (`"../../scripts/preflight.sh" --spec`).

**Out of scope**: `seed-board/SKILL.md:33`'s unrelated `!!`/`!` mention (about interpreting `preflight.sh`'s OWN log-line prefixes, not Claude command-substitution syntax — a false-positive match on a naive `!` grep, confirmed by reading the surrounding sentence) and `refine-task-ui/SKILL.md:57`'s markdown image syntax (`![alt](...)`, also a false-positive on a naive `!` grep) are NOT §7.5 sites and must not be touched. Any other plugin's skills (`peer-review`, `scaffold-project`) — zero hits, nothing to do there.

**New tests**: extend a capability-language-style test section (either `section-capability-language.sh` or a new `section-argument-semantics.sh` — dev agent's call, whichever reads more naturally given this is a different §7.x cluster from §7.1-§7.3's tool-naming pattern) asserting: (a) `check_absent` for the literal `!\`` byte-sequence in `build-next`/`implement-task`/`next-task`/`queue`/`seed-board`'s `SKILL.md` bodies (catches ANY `!`-prefixed command-substitution span, not just this one, in case a future skill adds another); (b) `check` that all 5 files carry the new explicit `Pre-start check — run this now, before anything else:` step with the `preflight.sh --spec` command intact; (c) `check_absent` for the literal `ARGUMENTS:` header in `ask-brain`/`ask-identity`'s bodies; (d) `check` that both files' new prose contains "remainder of the user's request" and preserves the role-name examples (`ask-identity`) / verbatim-question framing (`ask-brain`); (e) a repo-wide grep-based check (`grep -rln '!\`' plugins/*/skills/*/SKILL.md` returns empty, `grep -rln "ARGUMENTS:" plugins/*/skills/*/SKILL.md` returns empty) as a belt-and-suspenders sweep so a missed 6th site can't hide.

## CDX-014 — ui-options/neural-view graceful degradation (§7.6)

**§7.6 exact text**: "WHEN `CLAUDE_CODE_SESSION_ID` is unset THE SYSTEM SHALL render `ui-options` decision pages with the resume link omitted, never fabricated. WHEN Claude job metadata under `~/.claude/jobs` is absent THE SYSTEM SHALL render `neural-view` without live-session data rather than erroring, and SHALL label session counts by host once both are surfaced."

**Three separate sub-requirements, each in a different state of readiness — investigated, not assumed:**

1. **ui-options resume-link omission** — `ui-options/SKILL.md:23` ALREADY instructs the acting agent: fill `__SESSION_ID__` "with the value of `echo \"$CLAUDE_CODE_SESSION_ID\"` ... if empty, replace the whole resume sentence with nothing." There is NO script that renders `templates/ui-options.html` — the whole page-build step is manual (`cp` + hand-edit via the agent's own editing tool, per SKILL.md step 1.2), so this is prose-level behavior, not runtime code. "No error" is naturally satisfied too: `echo "$VAR"` on an unset var never errors in an interactive/agent-run shell (no `set -u` context here). **Verdict: functionally already compliant** — this sub-requirement needs a PINNING TEST (red because the assertion doesn't exist yet, not because the behavior is wrong), matching the prose-assertion convention CDX-010 through CDX-013 already established for this epic (check/check_absent against SKILL.md body text). No SKILL.md wording change expected unless the dev agent finds the existing phrasing doesn't literally satisfy "never fabricated" (it doesn't fabricate — it omits — so it should pass as-is).
2. **neural-view degradation when `~/.claude/jobs` is absent** — ALREADY implemented AND already tested: `discover_sessions()` in `scripts/neural-view.py` (~line 1179) checks `jobs_dir.is_dir()` and returns `[]` if absent (no exception path); `tests/section-neural-view-sessions.sh` (~line 45-51) already has a fixture ("no jobs dir at all") asserting `GET /sessions` yields `[]` on a fresh instance with no jobs directory. **Verdict: fully satisfied already, zero code or test change needed** — the dev agent should CONFIRM this (run the existing test, read the existing code) and cite it in the PR rather than re-implement or duplicate it.
3. **"SHALL label session counts by host once both are surfaced"** — **genuinely unimplemented, the real work of this task.** `discover_sessions()`'s returned session dicts currently have NO `host` field at all (only `repo`/`description`/`state`/`startedAt`); the frontend (`templates/neural-view.html`) has no host concept anywhere — `sessionCountFor(repo)` (line 2502) is a flat count, consumed at the 3D repo-label (`${sessN} LIVE`, line 1226) and the hover tooltip (`${sess} active session(s)`, line 2619). Since Codex-side session discovery doesn't exist yet (a later epic's work, if ever — no CDX task currently proposes reading a Codex-equivalent of `~/.claude/jobs`), "once both are surfaced" never literally triggers today; the acceptance criterion is about the MECHANISM existing and behaving correctly once it does, which is testable NOW with synthetic multi-host fixture data (a session list containing both `host:"claude"` and `host:"codex"` entries) even though only `host:"claude"` sessions are ever discovered live today.

**Implementation plan for #3**:
- **Backend** (`scripts/neural-view.py`): `discover_sessions()` adds `"host": "claude"` to every returned session dict (hardcoded literal — this discovery function only ever reads `~/.claude/jobs`, so every session it finds IS a Claude Code session; a future Codex-discovery function, if ever written, would tag its own results `"host": "codex"` and both would merge into one list before `GET /sessions` responds). Document this in the function's docstring as the seed point for future host expansion — do not build speculative Codex-discovery machinery now, just the `host` field and its consumption.
- **Frontend** (`templates/neural-view.html`): add a `sessionHostsFor(repo)` helper alongside the existing `sessionCountFor(repo)` — groups that repo's sessions by `host`, returns e.g. `{claude: 2}` or `{claude: 2, codex: 1}`. At both consumption sites (repo-label text, line ~1226; tooltip text, line ~2619): when the breakdown has exactly ONE distinct host, render EXACTLY as today (`${sessN} LIVE` / `${sess} active session(s)` — zero visible change, backward compatible, no clutter for the common single-host case); when it has MORE THAN ONE distinct host, append a parenthetical breakdown, e.g. `3 LIVE (2 claude, 1 codex)` / `3 active session(s) (2 claude, 1 codex)`. This directly implements "once both are surfaced" — the label only changes shape when there's something to disambiguate.

**Constraints to preserve verbatim**: the existing single-host label/tooltip text format (`${sessN} LIVE`, `${sess} active session(s)`) must be byte-identical when only one host is present — this is the overwhelmingly common case today and must not regress. `discover_sessions()`'s existing privacy invariant (never reads a job's `detail`/`output` fields) is unaffected — `host` is a new literal-constant field, not derived from any new data read.

**New tests**:
- ui-options: a new or extended section (e.g. `section-ui-options-degradation.sh`, or fold into an existing skill-contracts-style file — dev agent's call) asserting `ui-options/SKILL.md`'s body contains the resume-link-omit-when-empty instruction (pin the "if empty, replace the whole resume sentence with nothing" phrase or equivalent).
- neural-view absent-jobs-dir: confirm the EXISTING `section-neural-view-sessions.sh` assertion still passes (no new test needed, cite it).
- neural-view host-labeling: extend `section-neural-view-sessions.sh` (or a new section) with a fixture that seeds ≥2 synthetic `~/.claude/jobs/<id>/state.json` files (reusing this file's existing fixture-building helpers), asserts `GET /sessions` includes `"host":"claude"` on every entry, then — since there's no live Codex-discovery path to seed a real second host through the SAME code path — either (a) unit-test `sessionHostsFor()`/the label-building logic directly by loading `neural-view.html`'s script and feeding it a synthetic multi-host `window.__sessions` array (mirroring how other neural-view frontend tests in this repo already slice/eval inline `<script>` blocks — see the `anonymous-listener-slice-eval` pattern in the dev brain), confirming the parenthetical breakdown appears only when >1 host, or (b) note in the PR why frontend logic coverage was deferred if genuinely impractical within scope — dev agent's judgment call, but prefer (a), a real test beats a deferred one.
