## §8.6 cross-repo config sync — ADDED

- **§8.6.1 discovery** — WHEN `sync-configs.py` runs without `--repo` THE SYSTEM SHALL discover every immediate child of the scan base (`--scan`, else `~/Development`) carrying a `<child>/.claude/.neural-network` marker file, EXCLUDING the repo the script itself lives in (that repo evolves its own config through the build loop, not this script).

- **§8.6.2 versioned sync rules** — THE SYSTEM SHALL maintain an ordered, versioned list of sync rules, each with a stable id, a detector, and an applier, so new rules can ship incrementally without touching prior ones. Shipped rules: `strip-schema-data-key` (removes a literal top-level `$schema:` YAML data line duplicating the `# yaml-language-server:` comment header), `ensure-feedback-key` (adds `feedback: true` — or `false` via `--feedback-value` — to the end of the `methodology:` block when absent), `sw062-feedbacks-migration` (moves a legacy `.claude/feedback/` dir to `.claude/feedbacks/` and drops the corresponding `.gitignore` line, when the legacy dir exists and the new one does not).

- **§8.6.3 surgical edits** — All YAML edits SHALL be surgical line-level text operations preserving every other byte (comments, formatting); THE SYSTEM SHALL NOT parse-and-redump YAML.

- **§8.6.4 validation gates** — THE SYSTEM SHALL validate a target repo's config (via `validate-config.py`) BEFORE any edit — an already-INVALID config aborts that repo, reported, untouched — and AFTER edits — a rule that produces an invalid config SHALL be rolled back, with nothing committed.

- **§8.6.5 git safety protocol** — THE SYSTEM SHALL read the target repo's `project.mainBranch`. WHEN the repo's current branch equals that main branch AND its working tree is fully clean, edits/commit/push SHALL happen directly on the live checkout. OTHERWISE (non-main branch, or ANY dirty tree) THE SYSTEM SHALL NOT touch the live checkout or switch its branch: it SHALL apply changes in a temporary detached worktree created from `origin/<mainBranch>`, commit and push there (`git push origin HEAD:<mainBranch>`), then remove the worktree. Exactly the files the applied rules own SHALL be staged, nothing else.

- **§8.6.6 commit identity** — THE SYSTEM SHALL resolve the commit identity via `identity.sh orchestrator`, run with cwd inside the TARGET repo's working copy (per-clone resolution), and commit with those `-c` flags.

- **§8.6.7 dry-run default** — WITHOUT `--apply`, THE SYSTEM SHALL print per-repo rule decisions and diffs and change nothing, locally or remotely; `--apply` is required to write, commit, or push.

- **§8.6.8 report** — THE SYSTEM SHALL report, per repo: path, branch/route (`main` / `worktree` / `skipped-invalid` / `no-op` / `dry-run`), rules applied, pre/post validation results, commit sha, and push result; plus one final `AGGREGATE` line summarizing repos/synced/noop/skipped counts and mode.

## §9 Invariants — ADDED

- Cross-repo config sync (`sync-configs.py`) never edits or commits on a live checkout that is on a non-main branch or has a dirty tree — it always routes through a temporary worktree in that case.
- Cross-repo config sync never adds rules for config keys the plugin hasn't shipped yet.
