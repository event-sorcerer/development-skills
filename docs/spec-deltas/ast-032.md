## §8.2 — CLARIFIED (assembly order vs. note-wins recency)

§8.2 reads: "THE SYSTEM SHALL build the turn context as: systemPrompt persona + names +
roster (§11) + top-k recalled notes + rolling summary + last N turns (N≤6) + user
message, under a hard total budget (≤ ~6k tokens) with per-component caps; the rolling
summary SHALL be size-capped and refreshed every K turns."

Read literally, that prose lists recalled notes BEFORE the rolling summary. AST-032's
acceptance criterion is that the rolling summary "never contradicts a fresher note
in-prompt: note wins" — which requires the OPPOSITE ordering in the assembled prompt: a
recalled note is this turn's sharper, independently-ranked signal, while the summary is
a coarse digest that is only regenerated every K turns (so it can go stale for up to
K-1 turns). Placing the note LATER in the system text lets it win by prompt-order
recency without any explicit contradiction-detection logic, which this v1 does not
attempt (and is not asked to).

`turns.compose_context`'s `system_parts` assembly order is therefore:

    persona -> roster -> rolling summary -> recalled notes -> last N turns

(only summary and notes are swapped relative to §8.2's literal listing order; persona,
roster, and turns keep their documented positions). Per-component token/char budgets
and the clip precedence rules (§8.2's own "notes: rank-ordered prefix" / "summary: hard
char-cap" / "turns: oldest-first drop") are unaffected — this delta changes only WHERE
each already-clipped component's text lands in the concatenated system string, not how
much of it survives clipping.

Proposed EARS text, paste-ready for §8.2:

> §8.2 THE SYSTEM SHALL build the turn context as: systemPrompt persona + names + roster
> (§11) + rolling summary + top-k recalled notes + last N turns (N≤6) + user message,
> under a hard total budget (≤ ~6k tokens) with per-component caps; the rolling summary
> SHALL be size-capped and refreshed every K turns; recalled notes SHALL render AFTER
> the rolling summary so a note that contradicts a stale summary wins by prompt-order
> recency (note wins).

## Bookkeeping this task adds (not a spec amendment, an implementation note)

No new persistence or refresh mechanism: the rolling-summary refresh cadence (every
`turns.SUMMARY_REFRESH_EVERY_K_TURNS` turns, currently 8 — the design doc's mention of
"K=6" does not appear in the doc's own body and is not adopted here, since changing the
already-tested default is out of scope for an ordering fix) and its size cap already
live in `turns.py`'s `_advance_session_state`/`run_turn`, wired synchronously on the
request thread by `engine.py`'s `_chat` (`SessionStore.load_state` -> `run_turn` ->
`SessionStore.save_state`, per-root locked). `store.py` persists whatever
`session_state` it is given via its existing atomic tmp-file + `os.replace` write; it
does not gain a separate `refresh_summary` entry point, to avoid a second, divergent
refresh mechanism alongside the one `section-assistant-turns.sh` already tests.
