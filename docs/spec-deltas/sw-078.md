## §7.4 board.sh — AMENDED (local item-id cache; GraphQL economy)

§7.4 (task #13) specified pagination so `next`/`list`/`move`/seeding never silently truncate past one API page. It did not address a separate cost: every mutating command that needs an issue's project-item id (`move`, `prio`, `est`, `add`'s finish step, `adopt`) re-paginated the WHOLE board from page 1 to find that one id, every single time — on a 95-item board this made a routine `move` cost the same as a full `list`. Live evidence (issue #78): three board mutations in one session burned ~1400 GraphQL points this way, exhausting the hourly budget for every session sharing the account. `add`'s post-create visibility check made this worse, re-listing the whole board up to 3 times per filed issue.

THE SYSTEM SHALL maintain a local item-id cache, `.claude/board-cache.json` (gitignored), mapping `issue# -> {itemId, status}`:

1. **Cache-first resolution.** `move`, `prio`, `est`, `add`'s add-finish step, and `adopt` SHALL resolve an issue's project-item id from the cache before ever issuing a `gh` call. A cache HIT SHALL cost zero `gh` calls. A cache MISS SHALL cost exactly the one full item-list call that resolution would have required anyway; that call's full result SHALL replace the WHOLE cache (every issue on the board, not just the one being resolved) as a side effect, so every other issue's lookup in the same session is then a hit.
2. **Whole-board commands refresh, never read.** `next`, `list`, and `audit` are the only commands that genuinely need the whole board; they SHALL continue to paginate it in full (per §7.4's original pagination guarantee, unchanged) and SHALL refresh the cache as a side effect of that fetch, rather than ever reading from it — a whole-board command's own correctness SHALL NOT depend on the cache being present or fresh.
3. **Bounded, cache-aware visibility polling.** `add`'s and `adopt`'s post-create visibility check (waiting for `gh project item-add`'s eventually-consistent result to become visible) SHALL be capped at 2 attempts with backoff, using the cache-aware resolution above for each attempt (down from 3 attempts, itself down from a pre-#77 value of 10). Exhausting the cap without becoming visible SHALL defer to the #77 rate-limit queue (an `add-finish`/`move` op is queued) rather than retrying further or erroring.
4. **Cache invalidation on a rejected mutation.** WHEN a mutation's `gh project item-edit` call fails for a reason OTHER than a rate limit (e.g. the cached id no longer resolves to a live project item) THE SYSTEM SHALL drop that issue's cache entry and re-resolve it ONCE via a fresh full-board lookup, then retry the edit against the freshly-resolved id before surfacing a failure. This SHALL NOT apply when the failure is rate-limited (that case queues, per §7.7, and the cache entry is left untouched since the id itself was never shown to be wrong).
5. **Every successful mutation updates its own cache entry.** A `move` that succeeds SHALL update the cached `status` for that issue (so a later `move`/`prio`/`est` on the same issue in the same session is a cache hit); `prio`/`est` successes SHALL refresh the cached `itemId` without disturbing the cached `status` (they don't change it).
6. **`_current_status` (the flush stale-move-regression guard, §7.7/#92) is deliberately excluded from cache-first reads.** It always performs a live full-board fetch (refreshing the cache as a side effect, but never trusting it) because the very reason a move sits in the local rate-limit queue is that it never actually applied — the cache can only reflect mutations this script itself applied, never an out-of-band remote change or a still-pending queued one, so trusting it here would silently defeat the #92 regression guard.
7. **Best-effort writes.** Every cache write (put/drop/whole-board refresh) SHALL be best-effort: a failure to write (e.g. a read-only `.claude/`, a full disk) SHALL NOT fail the underlying move/prio/est/add/adopt (which has already succeeded against GitHub), and SHALL NOT leak a raw traceback. A lookup's correctness (finding an id that was just fetched) SHALL NOT depend on that fetch's cache-refresh write having succeeded.
8. **Gate fingerprint exclusion.** `.claude/board-cache.json` SHALL be excluded from `gate.sh`'s tree fingerprint (`tree-state.sh`), the same treatment as `.claude/telemetry.jsonl`/`.claude/lessons.jsonl` — a routine board mutation (by this lane or a concurrent one) must never invalidate an unrelated, still-current gate pass.

## §9 Invariants — AMENDED

- (from §7.4) restated: a mutating board command's item-id resolution SHALL NOT re-paginate the whole board when a valid cache entry exists; a cache miss SHALL cost exactly one full-board call, which SHALL refresh the whole cache.
