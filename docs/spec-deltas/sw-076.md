## §7.8 board-reflects enforcement — ADDED

Human directive: work must always be reflected on the GitHub board, enforced mechanically rather than by skill prose ("scripts decide, the model obeys").

WHEN `gh pr create` is invoked THE SYSTEM SHALL (via `guard-pr-create.sh`, a PreToolUse(Bash) hook mirroring `guard-board-move.sh`'s parsing discipline — SW-011: parse the real argv of the invocation, never substring-match the whole command line) BLOCK it unless the PR body references a board issue (`Closes #N` / `Fixes #N`, case-insensitive, or a qualified `<slug>#N`), with a remediation message directing the agent to file or pick the issue first via `board.sh`.

Two layers, in order, keep a heredoc BODY (text being WRITTEN to a file, e.g. via `cat > f <<EOF`) from ever being mistaken for a real invocation — the exact false-positive class SW-011 originally named, one level removed, since shlex has no concept of heredocs and flattens a body into the same token stream as everything else:
1. **Heredoc-range exclusion (review round 3 — round 1's position tracking alone was insufficient: a literal `&&`/`;` sitting INSIDE a heredoc body's text flips a position-only tracker's state just as a real operator would).** BEFORE tokenization, a quote-aware line scanner locates each unquoted `<<`/`<<-` operator and its delimiter (bare or quoted), then drops every line from just after the operator through the matching terminator line — word-for-word; `<<-` strips leading TABS (only tabs) from both the body and the terminator line before comparing, per bash semantics; quoting the delimiter changes nothing about what counts as the matching terminator. Those dropped lines never reach tokenization, so nothing inside them can be scanned, matched, or flip any tracking state — proven against `cat > notes.txt <<EOF` / `setup && gh pr create --body test` / `EOF`, which must NOT block. `<<<` here-strings (no body lines) are left alone. An unterminated heredoc fails closed as unparseable.
2. **Command-start-position tracking (review round 1).** Once heredoc bodies are gone, `gh pr create` SHALL count as a real invocation ONLY at a genuine command-start position — token 0 of the parsed command, or the token immediately after a `;`/`&&`/`||`/`|`/`&` operator (leading `VAR=value` assignments and an `env` prefix do not count as consuming that position) — so an argument to some OTHER command (e.g. `echo gh pr create`) is never mistaken for one.

KNOWN RESIDUAL GAPS (documented, not implied to be covered): the line scanner is a lightweight quote-tracker, not a full shell parser — a `<<` inside a `$(...)`/backtick command substitution is scanned the same as top-level text; a dynamically-built heredoc delimiter is not specially handled.

The body SHALL be sourced from `--body`'s value directly, or read from disk for `--body-file <path>` resolved against the hook process's actual working directory — folding in any leading `cd <dir>`/`pushd <dir>` found at a command-start position earlier in the SAME command string — NOT unconditionally against the repo root (review round 1: resolving against the root produced false blocks/allows whenever the session's cwd was a subdirectory); a `cd` target that is a shell variable, command substitution, or otherwise not statically resolvable is a documented residual gap and leaves the tracked directory unchanged rather than guessing. `--body-file -` (stdin) and the absence of both flags SHALL fail closed (blocked) — a PreToolUse hook cannot inspect stdin content, and there is nothing to check without either flag. A body that fails the reference check but contains an unexpanded command substitution (`$(...)` or a backtick) SHALL report that it cannot verify the body, rather than the generic missing-reference message. Shell-interpreter wrapping (`bash -c "..."`, nested) SHALL be unwrapped the same bounded-depth way `guard-board-move.sh` does; an unparseable invocation that still mentions `gh`/`pr`/`create` SHALL fail closed as unparseable. This guard SHALL NOT be registered in this plugin's `hooks/hooks.json` (unlike `guard-board-move.sh`, shipped to every consumer automatically) — it is opt-in per consumer repo via a documented `PreToolUse` snippet in the plugin README, added to that repo's own `.claude/settings.json`.

The same guard SHALL warn (never block) when the current branch does not match `project.branchPattern`, on every `gh pr create` invocation it evaluates.

`board.sh audit` (new subcommand) SHALL reconcile board reality against the repo and print one `DISCREPANCY:` line per finding, exiting 1 if any are found and 0 with a clean report otherwise:
1. Open PRs (`gh pr list --state open`) whose body contains no bare `#N` reference.
2. Local branches matching `project.branchPattern` with no board item in status `In progress` for the branch's embedded `<id>`.
3. Board items in status `In progress` with no matching local branch.
4. UNDER `work.type: local` ONLY: merged `project.mainBranch` commits (last 200, by `git log`) whose subject contains no `#N` reference, EXCLUDING a documented allowlist of recognized orchestrator process-commit classes — subjects starting with `retro(`, `spec(`, `feedback(`, or `config:`. Commits in this allowlist SHALL be exempt from being flagged as a discrepancy but SHALL always be counted and printed in the report (`recognized orchestrator process-commit classes: retro( (N), spec( (N), feedback( (N), config: (N)`) — never silently hidden. The 200-commit scan window itself SHALL be stated in the report UNCONDITIONALLY (`commit scan window: last 200 commits`), in both the clean and the discrepancy case (review round 1: a repo with more than 200 commits since the last audit must never see a bare "AUDIT: clean" that silently implies the whole history was checked).

`board.sh audit` reads board state via the existing `gh_project_items_json` plumbing (rate-limit queue applies, same as every other verb); the branch/commit checks are local-git-only and require no `gh` call.

## §9 Invariants — ADDED

- (from #76) `gh pr create` never lands without a board-issue reference in its body when `guard-pr-create.sh` is registered in the consumer repo's hooks — enforced by a parsed-argv check, not a skill instruction.
- (from #76) `board.sh audit` is the mechanical backstop for drift in BOTH directions (board items with no matching local reality, and local reality — PRs, branches, and, under `work.type: local`, merged commits — with no matching board reference); a clean run has exit code 0, any discrepancy has exit code 1, and recognized orchestrator process-commit classes are always enumerated, never hidden, even though they're exempt from the commit-subject check.
