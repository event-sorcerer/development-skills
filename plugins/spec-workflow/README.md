# spec-workflow

Spec-driven autonomous build workflow for Claude Code: a GitHub Project board is the source of truth, tasks are implemented TDD-first by delegated dev agents, a single gate command decides advancement, and humans steer asynchronously through issue comments and UI-option pages.

Everything project-specific lives in the consumer repo's **`.claude/project.yaml`** (versioned, `schemaVersion: 2`, YAML — [schema](./schemas/project-config.schema.json)): boards + field ids, one or more specs with epics/task ranges/dependency guards, hard invariants, gate/dev commands, `docs[]` sets (where documentation lives + which code paths each covers — one set for a standalone repo, one per package for a monorepo; the reviewer blocks PRs that change covered behavior without updating them), the `delegation.identities` roster (each role's git author templates AND its ALLOWED `models`, resolved per-clone; a role may be an array of per-package identities routed by `covers` globs), paths, and methodology knobs (`maxInProgress` — the concurrency knob: board WIP limit AND parallel implementation lanes, 1 = sequential; `iterativeUI`, `isolationSuite`, `autoMerge`/`mergeMethod` — autonomous PR review, approval, and merge in place of human approval; protocol in [`skills/build-next/references/auto-review.md`](./skills/build-next/references/auto-review.md) — and `feedback` (`true` shorthand for `{enabled: true, feed: .claude/feedbacks/feed.yaml, roles: [orchestrator], autoTriage: false}`) — structured per-iteration process feedback about the workflow itself, triaged into backlog/brain-note/graduate/upstream/ignore at retro time; mechanics in `scripts/feedback.py`. Also `work` (absent == `{ type: pr }`): `type: pr` (default, today's behavior) or `local` (no PR — review is `git diff <mainBranch>..<branch>`, approval recorded as an issue comment, the orchestrator squash-merges locally + pushes main); `sync.mode` (local only, absent == `realtime`) governs WHEN queued board mutations flush — `realtime`/`task-close`/`session-end`/`manual` — with `blocked`/`new-item` events always syncing immediately regardless of mode (the safety valve); resolved with defaults via `config.py get work.type`/`get work.sync.mode`, deferral decisions via `scripts/work-mode.sh should-sync <event>`; protocol in `skills/implement-task/SKILL.md`, `skills/build-next/SKILL.md`, and `references/auto-review.md` §5. The feed lives in `.claude/feedbacks/` (adjacent to `project.yaml`), a tracked, orchestrator-mediated archive — committed and pushed alongside code by default (opt out via the repo's own `.gitignore`), and never read or written by dev/reviewer subagents, same isolation as the identity brains). Refs in `evidence[]`/`routing.ref` are qualified with the emitting repo's own `project.name` (`<project.name>#N`) so a multi-project archive stays unambiguous — a foreign-qualified ref passes through verbatim, and the generalization ban on `summary`/`generalized` rejects qualified refs too, not just bare ones; `feedback.py migrate-qualify` is a one-shot, surgical, idempotent migration for a feed that predates this.

**Editor support**: `project.yaml`'s first line is a `# yaml-language-server: $schema=…` modeline; the Red Hat YAML extension (`redhat.vscode-yaml`) reads it for hover + autocomplete. [`schemas/project-config.schema.json`](./schemas/project-config.schema.json) is the **authoritative reference** for every key: every property and enum value carries a `description`/`enumDescriptions` (checked mechanically by `tests/section-schema-lint.sh`'s `schema-lint.py` walk — a future undocumented key fails the gate), so hovering any key in an editor shows its explanation, possible values, and default. `setup-project` also writes a `.vscode/` `yaml.schemas` mapping and an extension recommendation. **Legacy**: a `.claude/project.json` (schemaVersion 1, old `delegation.devModel/reviewModel/prReviewModel` keys) is still read — normalized to the v2 shape with a deprecation warning; `init-config.sh` converts it to `project.yaml`. Requires **PyYAML** (`pip3 install pyyaml`).

## Scripts (`scripts/`)

`config.py` (the shared loader: finds `.claude/project.yaml`, normalizes legacy json, `get`/`json`/`path`/`set` verbs for bash callers — `set` surgically edits one key, leaving comments/formatting intact) · `board.sh` (next/show/move/prio/est/add/bug/adopt/flush/ensure-labels/list/issues/comment/edit-body/fields/config/metrics/audit — `add [--type bug|feature|inbound]` files work into Backlog with the matching label; `bug` is a thin alias of `add --type bug`; `adopt <issue#>` adds an EXISTING issue to the board, idempotently, and sets its initial Status (matching `add`'s behavior — item-add itself never sets Status, so without this an adopted item would land with no status; #92); `ensure-labels` idempotently creates any configured label (`labels.bug`/`labels.feature`/`labels.inbound`) missing on the repo — run by `setup-project` and `seed-board.sh` so `add`/`bug` never fail on a label that was only ever declared in config, not created on GitHub; board/issue listing (`next`/`list`/`move`/`prio`/`est`/`issues`, and `seed-board.sh`'s own item enumeration) paginates past gh's per-call `--limit` ceiling via `paginate.sh`'s escalating-limit loop — no fixed 400/500-item truncation; **item-id cache (#78)** — `move`/`prio`/`est`/`add`/`adopt` resolve an issue's project-item id from a local cache (`.claude/board-cache.json`, gitignored: issue# -> {itemId, lastKnownStatus}) before ever re-paginating the whole board: a cache HIT costs zero gh calls, a cache MISS costs exactly the one full item-list call that would have happened anyway, which refreshes the WHOLE cache as a side effect (so every other issue's lookup that session is then free too); `next`/`list`/`audit` are the only whole-board reads and refresh the cache as a side effect rather than reading it; every successful mutation updates its own cache entry, and a mutation rejected for a reason other than a rate limit (e.g. the cached id no longer resolves) drops the entry and re-resolves ONCE against a fresh full-board lookup before retrying; `add`/`adopt`'s post-create visibility poll is capped at 2 attempts with backoff (was 3, and 10 pre-#77), each attempt itself cache-aware, before deferring to the #77 queue; the cache file is excluded from `gate.sh`'s tree fingerprint (`tree-state.sh`), same treatment as `telemetry.jsonl`/`lessons.jsonl`, so a routine mutation never invalidates an unrelated still-current gate pass; all cache writes are best-effort (a read-only `.claude/`, e.g., never fails the underlying move/prio/est/add, matching `telemetry.py`'s own write philosophy); **rate-limit resilience** — `move`/`prio`/`est`/`add`'s item-add step/`adopt` (including their item-id LOOKUP, not just the mutation itself), on a GitHub rate limit, append the op to `.claude/board-queue.jsonl` (gitignored) instead of failing, print `QUEUED (rate-limited until <reset>): <op>` (reset from `gh api rate_limit`, works even when GraphQL itself is exhausted), and exit 0; **detection is probe-based, not just text-matched** — a cheap case-insensitive "rate limit" substring check runs first, but gh sometimes masks GraphQL exhaustion as an unrelated error with no "rate limit" text at all (e.g. `unknown owner type`), so any gh-project failure that misses the text match probes `gh api rate_limit` and treats `.resources.graphql.remaining == 0` as ground truth for rate-limited, `> 0` as a real error surfaced verbatim (never silently swallowed into a queue); `flush` replays the queue in order — a queued `move` is skipped if the target status already holds, and a re-trip mid-replay re-queues the remainder and exits 0; **flush concurrency (#92)** — flush is guarded by an `mkdir`-based lockdir mutex next to the queue file (bash-3.2-portable: macOS has no `flock` builtin, `mkdir` is atomic on any POSIX filesystem) — a second flush that can't acquire the lock SKIPS immediately with a one-line message instead of blocking or replaying concurrently, and a lock older than `BOARD_QUEUE_LOCK_TTL` (default 600s) is broken with a warning so a crashed flusher can't wedge the queue forever; replay never blindly truncates the queue file — it atomically `mv`s it aside (same filesystem, a single rename) before replaying, then appends any re-queued remainder back onto the LIVE queue path, which new `queue_append`s may have grown meanwhile, so a concurrent append can never land in a snapshot-then-clear gap and get silently lost; before replaying, the moved-aside batch is deduped by `(op, issue)` for `move`/`prio`/`est` — keeping only the most-recently-enqueued entry (by `ts`, never rewritten across a requeue) — so a stale requeued op tail-appended behind a newer op for the same issue can never re-apply and regress a value the newer op already advanced past; `next`/`list`/`show` auto-flush a non-empty queue first, and fail fast with `RATE-LIMITED until <reset> — work continues; mutations queue; retry reads after reset.` on their own rate limit (masked or not) instead of a masked error; `list`/`issues` gate on gh's exit code so a gh failure never reaches an ungated JSON parse — see `board-queue.sh`; **`audit`** (#76) reconciles board reality vs the repo and prints one `DISCREPANCY:` line per finding — open PRs whose body has no `#N` reference, `project.branchPattern`-matching branches with no In-progress item, In-progress items with no matching local branch, and (`work.type: local` only) merged `mainBranch` commits — the last 200 (the window is stated in the report unconditionally: `commit scan window: last 200 commits`, so a clean report never silently implies the whole history was checked) — whose subject has no `#N` reference; a documented allowlist of recognized orchestrator process-commit prefixes (`retro(`, `spec(`, `feedback(`, `config:`) is exempt from that last check but always counted and printed — never silently hidden — via `audit.py`; ## Enforcement hooks

`guard-board-move.sh` ships in this plugin's `hooks/hooks.json`, so every consumer repo gets the gate-before-review guard automatically on install. `guard-pr-create.sh` (#76) does NOT — it's a more opinionated, blocking gate on `gh pr create` itself, so it's opt-in: add it as an additional `PreToolUse` hook in the consumer repo's own `.claude/settings.json` (or `.claude/settings.local.json`):

```json
{
    "hooks": {
        "PreToolUse": [
            {
                "matcher": "Bash",
                "hooks": [
                    {
                        "type": "command",
                        "command": "bash \"${CLAUDE_PLUGIN_ROOT}/scripts/guard-pr-create.sh\""
                    }
                ]
            }
        ]
    }
}
```

Run `board.sh audit` periodically (or at session/loop checkpoints) to catch drift the guards alone can't — board items and PRs/branches/commits that have already diverged, not just the next mutation.

## Identity brains

Each agent role (dev / reviewer / orchestrator, extensible) owns a **private** brain of atomic zettel notes under `.claude/identities/<role>/brain/` in the consumer repo. Brains give each identity durable memory that evolves **separately** — a hard product requirement. `ROLE.md` (per role) is the stable, human-owned instruction; starter templates ship in [`templates/identities/`](./templates/identities/).

- **Isolation** — only the orchestrator process reads or writes a brain. Subagents never see a brain path; recalled lessons reach them as pasted text in their brief. One role reads another's note only through a deliberate `consult`.
- **Retrieval** — `brain.sh recall <role> --paths ... --keywords ...` runs **spreading activation**: notes seeded by path-glob/tag match, activation flowing along `[[wikilink]]` edges (2 hops), emitted strongest-first within a token budget. Graduated notes stop being injected but still bridge links.
- **Evolution** — at each PR close the orchestrator runs a retro: `mint` notes in its own wording (strength bumps on re-mint), `prune` stale links, `graduate` proven lessons, regenerate `DIRECTORY.md`, commit as the orchestrator identity.
- **Contract** — every recall/consult appends to `<role>/brain/.activation.jsonl` (a frozen JSON-lines format a live viewer consumes). Link metadata (weight/fires/last) lives in `links.json`; notes stay clean markdown.

Full protocol: [`skills/build-next/references/brains.md`](./skills/build-next/references/brains.md).

## Human steering

- **Issue comments** — read before every task; scope changes are folded into the issue body and acknowledged.
- **Iterative UI mode** (default ON) — UI decisions go to the human via `docs/ui-options/<task-id>.html` (favorite selector + likeable aspect keywords + notes; "Copy selection" produces the comment to paste). Disable per-clone with `touch .claude/ITERATIVE_UI_OFF` or permanently with `methodology.iterativeUI: false`.
- **Checkpoint flag** — `touch .claude/CHECKPOINT` pauses the loop at the next safe boundary with a handoff.

## Telemetry

`.claude/telemetry.jsonl` (gitignored) is the loop's own performance record — SPEC §8.4. `board.sh move` and `gate.sh` append events automatically (best-effort: a write failure, e.g. a read-only `.claude`, never fails the move or the gate); `build-next` records `review-round`/`task-close` events at the orchestrator level (see its SKILL.md). Four record kinds, one JSON object per line: `transition` (`task`/`from`/`to`/`ts` — `from` is not tracked by `board.sh`, so metrics only uses `to`+`ts`), `gate` (`task`/`ok`/`ts` — `task` is the current branch name, since `gate.sh` has no task id in scope), `review-round` (`task`/`round`/`verdict`/`ts`), `task-close` (`task`/`estimate`/`ts`). `board.sh metrics` (→ `telemetry.py <root> metrics`) reports cycle time per status, first-try gate rate, rework (multi-round review) rate, and a per-task estimate-vs-actual listing; malformed lines are skipped and counted, never fatal. See `scripts/telemetry.py`'s module docstring for the exact schema.

## Neural view

The identities' **brains** (`.claude/identities/<role>/brain/`) are the workflow's memory: markdown notes wired by weighted links, plus an append-only activation log of every recall. `neural-view` is the human's window into them — a live, JARVIS-style HUD (`python3 scripts/neural-view.py start`, default `http://127.0.0.1:4748`) that draws every brain as a 3D neural cluster you navigate in space: notes are glowing neurons (size ∝ strength, graduated ones dimmed), links are synapses with a slow ambient pulse drifting source→target so the network reads as alive even at rest, and as recalls fire the page lights the neurons and sends a brighter pulse traveling the specific edge in real time — you watch a thought propagate. Top-left gauges are grouped **per repo** — one labeled section per anchored repo (`.claude/.neural-network`), each listing all three roles (orchestrator first, then alphabetical) with notes / avg strength / synapses; a role with zero notes still gets a dimmed row rather than vanishing, so a fresh repo's brains read as visibly separate from a mature one's instead of one shared trio. A bottom ticker streams the raw activation log, and clicking a neuron opens its note. The server is **read-only** and stdlib-only; the page makes zero external requests (the renderer is [three.js](https://threejs.org), **vendored** into `templates/vendor/` and served same-origin — see `templates/vendor/README.md` for the exact version/source/sha256 — no CDN, no build step), honors `prefers-reduced-motion`, and keeps WCAG AA text contrast. Tolerates absent brains (empty graph, "no brains yet" state) and gracefully degrades to a clear message (gauges/ticker still live) if WebGL is unavailable. Drive it with the `neural-view` skill (`start` / `stop` / `status`); `--port` and `--dir` override the defaults.

**Stale-server / zombie detection (SW-067).** `status` and `start` never report a bare STOPPED / "never bound to port" when the pidfile is missing or stale but the configured port is still occupied — they report **STALE**, naming the PID/command holding it when discoverable. The free-vs-held check is a plain stdlib bind attempt (no external tool required); the PID/command is optional enrichment via `lsof` (PID) and `ps` (full command line), degrading to "PID unknown"/"held by another process" when either is unavailable. `stop --force` stops the pidfile-tracked server if any, **and** kills a detected zombie holding the port — but only when that process's own command line contains `neural-view.py`; anything else is reported and left alone, since this is the one destructive path in an otherwise read-only tool. Plain `stop` (no `--force`) is unchanged.

**3D navigation.** Drag to orbit, wheel/pinch to zoom (toward the current focus), right-drag or shift-drag to pan, click a neuron (raycast) to inspect it, and the ⌂ button or double-clicking empty space resets the camera. The view auto-fits every constellation — the whole network, with margin — on boot (even before the first `/graph` resolves), on every reload, and on resize, until you take the wheel; the fit math is a pure, hermetically-tested function of the graph's bounding sphere, not a "looked right once" heuristic. Clicking (or keyboard-focusing and pressing Enter on) a brain row in the sidebar flies the camera to frame that brain's own cluster — instantly under `prefers-reduced-motion`, eased otherwise — falling back to the repo region for a brain with zero notes; this counts as taking the wheel, same as any manual orbit/pan/zoom.

**Multi-repo aggregation.** One server can show every spec-workflow repo on the machine as "constellations" arranged on a 3D fibonacci sphere — each repo is a labeled region containing its own role-clusters, with a top-bar `repos` counter and activation-log entries tagged `[repo] ROLE · event`. Discovery: a repo is included iff `<repo>/.claude/.neural-network` marker file exists in an immediate child of the scan base (`--scan`/`$NEURAL_VIEW_SCAN`, default `~/Development`); `--dir`/`$NEURAL_VIEW_DIR` is always included regardless of the marker. A discovered repo with no brains yet still gets an empty labeled region rather than being dropped. Graph node ids are `<repo>/<role>/<slug>`; `/note/<repo>/<role>/<slug>` addresses one; `/events` cursors carry a per-repo, per-role offset. With no `--dir`/`--scan`/env at all and an empty scan base, it falls back to the git root of cwd — the original single-repo behavior.

**Hover inspection.** Moving the pointer over any element (throttled raycast) opens a small tooltip — no click needed, and it never intercepts pointer events (orbit/pan/click all work exactly as before): a note names its slug, repo/role, tags and strength; a repo region/label shows its note/brain counts plus that repo's board state (below); a synapse names its source→target, weight, and fire count. The hovered note or synapse also gets a brief visual highlight, reusing the same glow mechanism recalls use.

**Per-repo project state.** `GET /projects` reports each discovered repo's GitHub Project board — status counts, and the titles currently `In progress`/`In review` — read via **this plugin's own `board.sh`** (never `gh project` directly) invoked with `cwd=<repo root>`, so it resolves that repo's own `.claude/project.yaml`. Results are cached server-side for `$NEURAL_VIEW_PROJECTS_TTL` seconds (default 60) and the subprocess is bounded by `$NEURAL_VIEW_BOARD_TIMEOUT` seconds (default 12), so a hung `gh` call never stalls the rest of the page. A repo with no board configured is omitted; a `gh`/network failure degrades to a dim "board unavailable" badge instead of erroring. The Brains panel gets a **Boards** section (status counts + truncated task titles per repo) and the top bar gets a subtle `boards N/M` (ok/total) indicator.

**Live sessions (best effort).** `GET /sessions` surfaces locally-discoverable Claude Code sessions from `~/.claude/jobs/<id>/state.json` — background-job orchestration metadata the harness itself writes (state/cwd/a short name label/timestamps). It is **never** the conversation transcript (that lives elsewhere and is never read); only `state`/`cwd`/`name`/`createdAt` are consulted. A job counts as active if its state is `working` or it was touched within the last `$NEURAL_VIEW_SESSION_RECENT_SECS` seconds (default 900); its `cwd` is matched to a discovered repo by path prefix so the repo label picks up a `· N LIVE` suffix and the repo tooltip gets a session count. `$NEURAL_VIEW_CLAUDE_DIR` overrides `~/.claude` (mainly for tests). If no `jobs/` directory is discoverable at all, this simply returns `[]`.

## Testing

```bash
bash tests/run-tests.sh        # hermetic: validator fixtures, picker, preflight, init-config (fake gh)
python3 tests/check-evals.py   # eval case structure
claude plugin eval .           # skill evals on real models (early access)
```

`run-tests.sh` is a thin runner: it defines the shared `check`/`check_rc`/`check_absent`/
`lifecycle_start` helpers (in `tests/_lib.sh`) and sources every `tests/section-*.sh` file
in a fixed order. Each section file owns one area (config, gate enforcement, neural-view,
brain, etc.) and is sourced, not run standalone — it assumes `_lib.sh` and the runner's
shared state (`HERE`/`PLUGIN`/`FIX`/`fails`/`flaky`) are already in scope. To add a test to
an existing area, add a `check ...` line to that area's `section-*.sh`; to add a new area,
create a new `section-<area>.sh` (with the standard sourced-not-standalone header comment)
and register it in `run-tests.sh`'s `SECTIONS` array. This split keeps parallel build-loop
lanes from all appending to one file (a guaranteed rebase conflict on every merge).

Fake-`gh` fixtures that simulate a `gh` failure (masked rate limits, honest rate limits,
etc.) source their trigger text from `tests/fixtures/gh-failures/` — a corpus of REAL
captured `gh` outputs with provenance — rather than inlining invented strings beside the
classifier they test (see that directory's `README.md`). Whenever a masked or odd `gh`
error is observed live in any session, capture the raw bytes there immediately.
